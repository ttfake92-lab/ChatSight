## 一、这次真正要解决的三个问题

### 1. 主题能力

目标不是“加一个手动开关”，而是：

- 应用默认跟随系统深色 / 浅色主题
- 系统主题切换后，应用界面无需重启即可同步变化
- 现有深色变量已经定义在 `src/index.css`，优先复用，不要重做整套样式系统

当前观察：

- `src/index.css` 已经有 `.dark` 变量
- 但项目里没有任何地方根据系统主题去给根节点切换 `.dark`
- `src/App.tsx` 外层仍是固定浅色渐变背景：`from-blue-50 to-indigo-100`

建议的最小实现：

1. 在前端启动时读取 `window.matchMedia('(prefers-color-scheme: dark)')`
2. 根据结果给 `document.documentElement` 增删 `dark`
3. 监听系统主题变化并实时同步
4. 把 `App.tsx` 中硬编码的浅色背景改成能随主题工作的样式

验收标准：

- macOS 系统切到深色时，应用自动进入深色
- 系统切回浅色时，应用自动恢复
- 卡片、弹窗、文字、边框、背景都没有明显“亮块”残留
- 不要求本轮加入“手动主题偏好”或“三态主题配置”

## 二、打开软件不稳定：优先检查启动链路，而不是先补 UI

用户现象：

- 软件启动后经常出现“无法加载”

当前代码里最值得优先怀疑的点：

### A. 主窗口创建被 `wechat-cli init` 阻塞

`electron/main.ts` 当前流程是：

1. `app.whenReady()`
2. `await wechatExecutor.init()`
3. `createWindow()`

`wechat-cli` 默认超时 30 秒。  
如果初始化慢、失败或环境异常，窗口会延迟很久才创建，用户感知上就是“软件打不开 / 没加载出来”。

### B. 渲染进程启动后立即拉取会话，但缺少“可恢复”的初始化状态

`src/App.tsx` 首次渲染后立刻：

- `loadConfig()`
- `fetchSessions()`
- `loadSkills()`

如果主进程还没准备好，或者 `wechat-cli` 初始化失败，`fetchSessions()` 只会把错误写进 store；没有明确的初始化状态、重试路径或启动页引导。

### C. 当前“能打开窗口”和“能读到聊天数据”被揉成了一件事

这会让一个外部依赖故障拖垮整个产品第一印象。  
更合理的边界是：

- 应用窗口应尽快出现
- 微信数据源初始化可以异步进行
- 初始化失败时，给出明确状态和重试入口，而不是让用户面对模糊的“加载失败”

建议的最小实现方向：

1. 先创建窗口，再异步初始化 `wechat-cli`
2. 为初始化状态增加清晰建模：`idle / initializing / ready / error`
3. 渲染层只有在 `ready` 后再拉取 sessions；`error` 时展示可重试 UI
4. 给启动失败留出有意义的错误信息，最好能区分：
   - 未安装 `wechat-cli`
   - 未初始化
   - 命令超时
   - 其他执行失败

验收标准：

- 即使 `wechat-cli` 不可用，窗口也能稳定打开
- 用户能看到明确失败原因，而不是空白或无尽 loading
- 初始化失败后可以重试，不必重启应用
- 正常环境下启动时间不应被 `wechat-cli init` 阻塞

## 三、AI 摘要历史不能丢：把它从“组件瞬时状态”提升为“按会话存储的业务状态”

当前观察：

- `src/components/AISummary.tsx` 里用的是本地 `useState`
- 摘要没有和 `selectedSession` 绑定
- 没有持久化
- 没有“历史摘要”概念

这意味着当前实现天然无法满足：

- 切换会话后保留每个会话自己的摘要
- 重新打开应用后仍能看到之前生成过的摘要

建议的最小实现：

1. 新增一个摘要 store，例如 `summaryStore`
2. 以 `session.id` 或稳定的 `session.name` 作为 key 保存摘要
3. 至少持久化：
   - `summary`
   - `generatedAt`
   - 可选：`messageCount` 或 `sourceRange`
4. `AISummary` 根据当前会话读取 / 写入对应摘要
5. 切换会话时：
   - 有历史摘要则展示该会话自己的摘要
   - 没有历史摘要则展示空状态

验收标准：

- A 群生成摘要后，切到 B 群，再切回 A 群，A 的摘要还在
- A、B 两个会话的摘要互不串线
- 重启应用后，历史摘要仍可读取
- 如果选择的是全新会话，不能展示上一个会话的旧摘要

非目标：

- 本轮不要求做“多版本摘要历史”
- 本轮不要求做云同步
- 本轮不要求做复杂失效策略

## 五、推荐执行顺序

```text
1. 先修启动链路
   → 验证：窗口可稳定出现；异常环境有明确错误与重试

2. 再做摘要持久化
   → 验证：跨会话、跨重启都能保留且不串线

3. 最后接入系统主题
   → 验证：系统切换时应用实时跟随
```

理由：

- 启动稳定性是基础体验，优先级最高
- 摘要持久化会触及状态结构，适合在主题之前完成
- 主题改动相对独立，最后做最不容易互相干扰

## 七、实现时最容易犯的三个错

1. **把“跟随系统主题”做成手动切换开关**
   - 这会偏题。用户要的是随系统变化。

2. **只把摘要塞进 localStorage，却不按会话区分**
   - 这会把“丢失”修成“串线”，只是换一种坏法。

3. **只给加载失败加一句文案，却不拆开启动依赖**
   - 这不能真正解决“打开软件不稳定”。
